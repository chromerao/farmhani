# RAG 품질 개선 구현 계획 (Backend)

데이터(DB, Chunking 등) 및 프론트엔드/API 명세 변경 없이, 현재 백엔드의 LangGraph 파이프라인과 RAG 검색 로직만을 수정하여 응답 품질을 극대화하는 개선 계획입니다.

## 주요 변경 목표
1. 검색에 방해되는 불필요한 컨텍스트 압축 (Query Transformation)
2. 하이브리드 검색 점수 결합 방식 고도화 (RRF 알고리즘)
3. 무관한 문서 필터링 (Self-RAG Grading/Reranking)
4. 답변 생성 과정에 사용자 공개용 근거 요약 추가

---

## 1. 백엔드 RAG 파이프라인 (LangGraph) 변경 사항

**수정 대상 파일:** `backend/app/services/rag/pipeline.py`

### 1-1. `build_retrieval_query` 노드 (Query Transformation 도입)
- **현재 로직:** 식물 정보, 사용자 질문, 사진 분석 결과, 맥락을 하나의 긴 문자열로 결합하여 쿼리로 사용.
- **개선 로직:** OpenAI LLM을 사용하여 문맥을 파악하고, "벡터 검색에 최적화된 독립적인 핵심 키워드/질문 2~3개" 형태의 쿼리 리스트로 변환(Multi-Query). 이를 통해 노이즈 토큰으로 인한 검색 정확도 하락 방지.

### 1-2. `grade_or_rerank` 노드 (문서 필터링 구현)
- **현재 로직:** 입력받은 문서를 필터링 없이 그대로 다음 노드로 반환 (Dummy 처리).
- **개선 로직:** 
  - `retrieve_docs`에서 가져온 다수의 후보 문서(예: 상위 8개)에 대해 OpenAI 모델(또는 자체 스코어링 로직)을 거칩니다.
  - 문서가 사용자의 질문과 관련이 있는지 평가(Grade)하고 관련이 적은 문서는 제외합니다.
  - 가장 관련성 높은 3~4개의 문서만 최종적으로 추려내어 할루시네이션을 최소화합니다.

### 1-3. `generate_answer` 노드 (근거 요약 추가)
- **현재 로직:** 곧바로 `summary`, `possibleCauses`, `todayActions` 필드를 포함한 JSON을 출력하도록 프롬프트 지시.
- **개선 로직:** JSON 스키마 첫 번째 요소에 `"evidenceNotes"` 필드를 추가하도록 시스템 프롬프트를 변경.
  - 내부 추론 과정이 아니라 사용자에게 공개 가능한 짧은 근거 요약만 작성하도록 제한.
  - 이를 통해 최종 도출되는 답변의 근거성을 높이되, 클라이언트 응답 및 화면 렌더링에는 영향이 없도록 설계.

## 2. 하이브리드 검색 결합 알고리즘 변경 사항

**수정 대상 파일:** `backend/app/services/rag/vectorstore.py`

### 2-1. `merge_results` 함수 (RRF 도입)
- **현재 로직:** 키워드 검색 점수와 벡터 유사도 점수 중 단순 최대값(`max`)을 적용하여 결합. 점수 스케일의 차이로 인해 정확한 결합이 어려움.
- **개선 로직:** **Reciprocal Rank Fusion (RRF)** 알고리즘 도입.
  - 검색된 결과의 '순위(Rank)'를 기반으로 점수를 재산정.
  - 수식: `1 / (rank + 60)`
  - 벡터 검색과 키워드 검색 모두에서 높은 순위를 기록한 문서가 확실하게 상단으로 올라오도록 보정.

### 2-2. `search_documents` 함수 파라미터 조정
- Reranking 단계에서 충분한 풀(Pool)을 확보할 수 있도록, 1차 검색(Retrieval) 시 가져오는 후보 문서의 개수(`top_k`)를 기존보다 넉넉하게 상향 조정 (예: 4개 -> 8개).

---

## 기대 효과
- **데이터 구조나 프론트엔드를 수정하지 않으면서** 기존 코드를 최적화하여 답변 정확도 극대화.
- 불필요한 문서 배제로 엉뚱한 처방을 내리는 할루시네이션(환각 현상) 감소.
- 복합적인 증상이나 모호한 질문에도 더 논리적이고 정교한 단계별 가이드 생성.
