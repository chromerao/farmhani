# RAG 품질 개선 완료 보고서

지시하신 대로 프론트엔드 및 데이터 구조의 변경 없이 백엔드의 LangGraph 파이프라인과 RAG 검색 모듈만을 수정하여 품질 개선 작업을 완료했습니다. 

## 1. 구현된 주요 변경 사항

### 1) Query Transformation (쿼리 최적화)
- **변경 파일:** `backend/app/services/rag/pipeline.py` 의 `build_retrieval_query`
- **구현 내용:** 기존에는 사용자의 질문, 식물 종류, 사진 분석 결과 등 모든 정보를 긴 문자열로 묶어서 그대로 검색 엔진에 넘겼습니다. 이를 개선하기 위해 **OpenAI API를 호출하여 긴 문맥을 분석하고, 벡터 검색에 가장 유리한 독립적 키워드 3개를 추출**하도록 로직을 추가했습니다.
- **효과:** 불필요한 조사나 노이즈 단어로 인해 엉뚱한 문서가 검색되는 현상(희석 현상)을 방지합니다.

### 2) RRF (Reciprocal Rank Fusion) 적용
- **변경 파일:** `backend/app/services/rag/vectorstore.py` 의 `merge_results`
- **구현 내용:** 키워드 검색 점수와 벡터 유사도 점수는 스케일이 달라서 단순 최대값(`max()`)으로 합치면 한 쪽에 편향됩니다. 이를 해결하기 위해 두 검색 결과의 **순위(Rank)를 기반으로 점수를 합산하는 RRF 알고리즘(`1 / (rank + 60)`)을 도입**했습니다.
- **효과:** 두 가지 검색 방식에서 모두 상위권에 속한 문서가 최종 검색 결과의 최상단에 안정적으로 배치됩니다.

### 3) Self-RAG (리랭킹 및 필터링) 노드 구현
- **변경 파일:** `pipeline.py` 의 `grade_or_rerank`, `vectorstore.py` 의 `top_k` 수정
- **구현 내용:** 
  - 먼저 `search_documents`에서 불러오는 후보 문서의 수를 기존 4개에서 **8개**로 늘렸습니다.
  - 이후 `grade_or_rerank` 노드에서 OpenAI API를 사용하여, 각 후보 문서가 **사용자의 질문에 답변하는 데 실제로 유용한지(Yes/No)**를 자체 평가(Self-RAG)하도록 구현했습니다.
  - 관련성이 확인된 문서 상위 4개만을 추려내어 다음 노드로 넘깁니다.
- **효과:** 완전히 무관한 문서가 참조되어 엉뚱한 대답을 내놓는 '할루시네이션(환각)'을 강력하게 필터링합니다.

### 4) 근거 요약 프롬프트 적용
- **변경 파일:** `pipeline.py` 의 `generate_answer`
- **구현 내용:** 답변을 생성하는 LLM의 JSON 포맷 첫 번째 필드로 `"evidenceNotes"` 를 요청하도록 조정했습니다.
- **효과:** AI가 내부 추론 과정을 노출하지 않고, 사용자에게 보여줄 수 있는 짧은 근거 요약을 남기게 됩니다. 클라이언트에게는 기존과 동일한 형태의 정보만 활용되어 UI 깨짐이 없습니다.

## 2. 요약 및 확인 방법

작업된 코드는 기존의 `main.py`나 `app/api` 등의 구조를 건드리지 않으며, 외부 인터페이스(API Contract)를 변경하지 않도록 캡슐화되어 반영되었습니다. 

백엔드 서버 구동 시 내부적으로 다음과 같은 효과를 확인하실 수 있습니다:
1. 검색이 훨씬 빠르고 정확해진 점
2. 엉뚱한 문서가 Reranking 노드에서 걸러지는 점
3. 답변이 질문의 핵심(예: 특정 증상)에 맞게 논리적인 추론과 함께 제공되는 점
