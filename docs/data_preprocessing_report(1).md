### [Step 1] 1차 전처리 및 데이터 정규화 (Data Normalization & Formatting)

**[핵심 명제] 1차 데이터 전처리 및 정규화(Step 1)는 파편화된 5대 이종 데이터를 하나의 통일된 의미론적 표준 스키마(Standardized JSONL)로 통합 변환하고 불필요한 텍스트 노이즈를 제거하여 RAG 검색 엔진의 입력 데이터 무결성을 확보하는 공정입니다.**

#### 🧠 CoT (생각의 흐름) 기반 논리적 이유 서술
이종의 원시 데이터(Raw CSV)는 각기 다른 목적을 지니고 작성되었기 때문에 필드의 개수, 명칭, 데이터 타입이 파편화되어 있습니다. RAG 시스템에서 정보를 효과적으로 검색하기 위해선 우선 모든 데이터 형식을 하나의 통일된 의미론적 뼈대(Standard Schema)로 전환해야 합니다. 만약 이종 포맷의 정합성이 결여된 상태로 RAG 검색 엔진에 전달될 경우, 특정 필드의 편향 현상으로 인해 유사도 계산의 편차가 왜곡될 수 있습니다. 

따라서 1차 전처리의 핵심 추론 과정은 다음과 같습니다.
첫째, 각 데이터 파일의 고유한 컬럼 정보를 수집하여 통합 RAG 파이프라인의 명세 규칙(Standardized JSONL schema)으로 맵핑합니다. 이때 수집 주소(`source_url`), 라이선스 정보, 그리고 수집 타겟 식물명 리스트(`crop_or_plant`)를 구조적으로 추출하여 인덱싱합니다.
둘째, 데이터 내 섞여 있는 불필요한 HTML 엔티티, 줄바꿈 찌꺼기, 여러 개의 공백(Double Whitespaces) 등을 정규표현식(Regex Cleaner)을 통해 정제합니다. 텍스트 내부 노이즈가 제거되지 않으면 임베딩 벡터 인코딩 단계에서 문장의 원래 의미가 아닌 공백이나 양식 문자에 가중치가 할당되는 벡터 공간 왜곡 문제가 발생하기 때문입니다.
셋째, 검색 정확도를 높이기 위해 식물 사전 라이브러리를 연동하여, 임베딩 본문에 등장하는 60종의 타겟 식물 키워드를 메타데이터 태그로 강제 주입하는 필터링 메커니즘을 설계합니다. 이는 최종 Retrieval 단계에서 메타데이터 필터(Metadata Filtering)를 연동할 수 있게 함으로써, 엉뚱한 식물의 병해충 지식을 혼동하여 대답하는 고질적인 거대언어모델(LLM)의 환각(Hallucination) 에러를 방지하는 최적의 사전 예방 기법입니다.

#### 📊 1차 정규화 통합 데이터 구조 명세 (`rag_documents.normalized.jsonl`)
*   **출력 경로**: `data/interim/rag_documents.normalized.jsonl`
*   **통합 스키마 구조**:
    ```json
    {
      "source_id": "UUID (출처별 해시 식별값)",
      "source_key": "nongsaro_crop_tech | national_botanic_garden 등",
      "title": "문서 고유 타이틀",
      "text": "1차 클렌징 및 필드 결합이 완료된 RAG 임베딩 대상 원문 본문",
      "category": "indoor_care | crop_care | pesticide_safety 등",
      "collected_at": "YYYY-MM-DD",
      "source_url": "제공처 원본 URL",
      "license": "라이선스 명세",
      "crop_or_plant": ["몬스테라"],
      "safety_tags": ["not_diagnosis", "expert_check_required"]
    }
    ```

### [Step 2] 문맥 보존형 단락 분할 (Context-Aware Document Chunking)

**[핵심 명제] 문맥 보존형 단락 분할(Step 2)은 정규화된 문서 텍스트를 LLM의 토큰 한계와 검색 유사도 가중치 정밀도를 고려해 문장 경계를 보존하는 500자 단락 크기 및 50자 중첩 구간(Overlap)의 최적 청크(Chunk) 단위로 쪼개는 공정입니다.**

#### 🧠 CoT (생각의 흐름) 기반 논리적 이유 서술
아무리 1차 정규화가 완료된 본문이라 할지라도, 문서의 길이가 너무 길면 하나의 긴 텍스트 덩어리를 벡터 공간상에 인코딩해야 합니다. 이 경우 정보의 희석(Information Dilution) 현상이 발생합니다. 즉, 문장의 개별적인 세부 사실(예: 특정 해충 방제 희석 배수 등)이 문서 전체의 거시적인 맥락에 묻혀 흐릿해지는 것입니다. 따라서 본문을 적당한 크기로 쪼개어 검색 정밀도를 높여야 합니다.

그러나 청크의 길이를 지나치게 작게 쪼개면(예: 단어 단위 분할) 이번에는 주변 맥락(Context)을 잃어버리는 정보 단절 문제가 나타납니다. 이에 따라 파이프라인 엔지니어링 설계 시 **"청크 사이즈 500자(Chunk Size 500)"** 및 **"오버랩 50자(Overlap 50)"**라는 최적의 트레이드오프 파라미터를 추론하여 적용합니다. 500자는 단일 기상 시나리오나 단일 약제 안전 수칙을 의미 단위로 전달하기에 가장 적정하게 압축된 토큰 범위입니다.
또한, 문맥 손실을 방지하기 위한 분할 규칙(Sentence Boundary Tracker)을 구현합니다. 단순히 글자 수 500자로 정확히 반을 뚝 자르면 하나의 온전한 문장이 중간에 잘려 주어와 서술어가 깨지는 파편화 현상이 일어나므로, 마침표(`.`)나 줄바꿈(`\n`) 같은 자연스러운 문맥 경계 지점을 탐색하여 그 직전 지점에서 문단을 쪼개도록 강제합니다.
그리고 연속된 두 청크 사이에는 50자의 중첩 구간(Overlap)을 유지합니다. 이는 앞 단락과 뒷 단락 사이의 의미론적 교량 역할을 하여, 분할 지점 부근에 걸쳐 있는 정보도 RAG가 누락 없이 정확한 유사도를 계산하여 검색(Retrieval)할 수 있도록 보장하는 문맥 연속성 유치 메커니즘을 제공합니다.

#### 📊 청킹 완료 데이터 스펙 (`rag_chunks.sample.jsonl`)
*   **출력 경로**: `data/processed/rag_chunks.sample.jsonl`
*   **총 청크 개수**: 600개 (5대 원시 데이터 500건 + 국립수목원 100건)
*   **특징**: 의미론적 경계가 훼손되지 않은 문맥 보존형 청크 분할 완료.

### [Step 3] 고밀도 벡터 인코딩 및 차원 정렬 (Dense Vector Embedding & Alignment)

**[핵심 명제] 밀밀도 벡터 인코딩 및 차원 정렬(Step 3)은 600개의 정제된 단락 텍스트를 OpenAI `text-embedding-3-small` 모델을 이용해 1536차원의 고차원 실수형 벡터 공간 좌표로 투사하여 코사인 유사도 검색을 활성화하는 최종 물리 인코딩 공정입니다.**

#### 🧠 CoT (생각의 흐름) 기반 논리적 이유 서술
청킹된 600개의 한국어 텍스트 조각들은 사람이 읽을 수 있는 문자열 상태입니다. 검색 엔진이 이 조각들의 의미적 유사성(Semantic Similarity)을 판별하기 위해서는 텍스트를 기하학적 다차원 의미 공간의 실수형 좌표로 변환해 주어야 합니다. 

본 파이프라인에서 채택한 인코딩 모델인 OpenAI의 `text-embedding-3-small`은 한국어 및 다국어 지식 표현력이 극대화된 1536차원의 고차원 고밀도 공간 임베딩 모델입니다. 이 인코딩 과정을 통해 텍스트 내의 숨겨진 의미적 관계(예: "몬스테라"와 "찢잎" 간의 연관성 등)가 수학적인 실수 배열로 정교하게 변환됩니다. 두 청크 간의 거리가 가까울수록 코사인 유사도(Cosine Similarity) 값이 1에 수렴하게 되어 정확한 검색이 이루어집니다.
보안 및 인프라의 강건성을 보장하기 위한 추론 프로세스는 다음과 같습니다.
첫째, 실제 OpenAI 서버에 네트워크 요청을 넣어 실시간 연동 임베딩을 구성하기 전, 개발 단계에서의 가상 무결성 연동을 위해 로컬 해시 모드(`--mode hash`)로 모의 1536차원 벡터를 미리 빌드해 볼 수 있는 샌드박스 파이프라인을 구축합니다. 이를 통해 외부 API 호출 없이 로컬 단에서 데이터의 스키마 구조와 정합성을 1차적으로 확인합니다.
둘째, 실제 적재 시에는 로컬 캐시 파일인 `rag_chunks.embedded.jsonl`을 완전하게 빌드한 뒤, 원격 Supabase Vector DB에 로드하는 단계를 분리하여 설계합니다. 데이터의 가공과 DB 적재 행위를 물리적으로 분리(Decoupling)함으로써, 만약 업로드 중 네트워크 에러나 API 크레딧 초과와 같은 런타임 오류가 발생하더라도 이전에 잘 유지되고 있던 Supabase 내부의 기존 RAG 데이터를 그대로 유지하고 롤백할 수 있는 트랜잭션 무결성을 설계하였습니다.
